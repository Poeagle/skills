# iCost 个人消费分析系统

## 核心逻辑：预算对比法

不再按「必要/正常/浪费」分类，而是：

> **设定预算 → 对比实际 → 计算超支**

### 用户设定的预算标准

| 餐别 | 预算上限 | 工作日 | 周末 |
|------|---------|--------|------|
| 早餐 | ¥5 | 自费 ✓ | 自费 ✓ |
| 午餐 | ¥10 | **公司管饭，免费** 🏢 | 自费 ✓ |
| 晚餐 | ¥10 | 自费 ✓ | 自费 ✓ |
| **每天自费合计** | | **¥15**（早+晚） | **¥25**（早+午+晚） |

**计算逻辑（report.py 中的 budget_per_day）：**
- 工作日（Mon-Fri）：`return 15`（早¥5 + 晚¥10，午餐公司免费）
- 周末（Sat-Sun）：`return 25`（早¥5 + 午¥10 + 晚¥10）
- 早餐预算已含在总预算中，但 iCost 无早餐单独记录，追踪时只对比午+晚实际支出

**⚠️ 踩坑历史：**
- ❌ 一开始按所有外食做「必要/正常/浪费」分类 → 被用户否定（数字对不上、主观判断不准）
- ❌ 用了 ¥15/顿的自炊标准算节省 → 用户说没理解「工作日午餐免费」的前提
- ❌ 忘了加早餐预算 → 用户暴怒
- ❌ 描述和代码不一致：SKILL.md 写"微信推送"但 report.py 的 send_message() 硬编码 QQ bot → 被用户纠正。**任何时候加载此 skill，先读 report.py 确认 send_message() 的实际推送目标，不要信任 SKILL.md 的描述文字。**
- ❌ 用户说"需要改一下"时直接动手改成了我以为的改法——实际用户的意思是修不一致的描述，不是接受我提的方案。**先反问确认再动手。**
- ✅ 最终方案：直接设预算线，对比实际支出，算超出金额

## 报告逻辑

### 上周
- 消费总览：按一级分类拆分
- 伙食预算对比：实际 vs 预算（工作日¥15、周末¥25）→ 超出/节余
- 列出单餐超¥10的明细项

### 本月
- 总消费 + 日均
- 伙食超支总额（实际 - 天数×预算）
  - 预算 = 工作日数×¥15 + 周末天数×¥25
- 每周拆分对比（按自然周 Mon-Sun）
  - 周范围 M/D-D  实际¥XXX  预算¥XXX  🔺超/✅省

### 零食饮料
- 单独列出，视为「不在预算内的额外支出」

### 总结
- 伙食控制：省 ¥XXX
- 零食不买：省 ¥XXX
- 理想月支出

```
📋 消费周报

消费 ¥XXX（餐饮¥XXX · 日用¥XXX · 软件¥XXX）
伙食 ¥XXX，预算 ¥XXX，超出 ¥XXX
超标项：M/D¥XX备注 · M/D¥XX备注

────────────────────
📆 本月 1-D（X 天）
总消费 ¥XXXX（¥XXX/天）
伙食超支 ¥XXX（实际 ¥XXX - 预算 ¥XXX）

────────────────────
📊 每周伙食对比（工作日¥15/天 · 周末¥25/天 · 含早¥5）
第1周 5/1-5/7  实际¥55  预算¥125  ✅省¥70
第2周 5/8-5/14  实际¥246  预算¥125  🔺超¥121
...

────────────────────
🍿 零食饮料 ¥XXX — 不在预算内，额外支出

────────────────────
✅ 如果零浪费：原本 ¥XXXX/月
  伙食控制在预算内 → 省 ¥XXX
  不买零食 → 省 ¥XXX
  每月只需 ¥XXXX（¥XXX/天）

────────────────────
📌 资产更新了没？
```

### 格式死律
- 分隔线用 `─` x 20
- 数字用 ¥ 前缀，不要加"元"字
- 超标项用 `·` 分隔，不要换行（一条消息不要太长）
- 每周都要有上周数据，不能只报本月
- 末尾必须加资产提醒

## 工作流

### 每周用户操作
周日从 iCost 导出增量 xlsx（上周新记录），保持 iCost 默认命名（iCost_YYYYMMDD*.xlsx），放入 `~/records/finance/`

### 周一 07:00 自动流程
1. **扫描新 xlsx** — 对比 processed.log，有则调用 ingest.py 转换
2. **漏传检测** — 无新文件且最后记录不在上周 → QQ 提醒
3. **分析** — 读年度 csv，按预算规则计算
4. **推送** — 按固定格式生成报告，发 QQ

### 年度查询模式

用户问「去年全年花了多少」「统计一下去年全年」时：

1. 运行 `python3 ~/.claude/skills/icost-analysis/scripts/yearly.py [年份]`
2. 输出包含：全年总支出 · 一级分类拆解 · 每月趋势（含条形图） · 外食/零食/订阅独立统计 · 大额支出明细（≥¥300）
3. 注意区分一次性大件（MacBook、耳机等）和日常支出——去掉大件后算月均实用参考

### 行为模式分析

用户问「根据消费能分析出什么模式」「消费有什么规律」「从数据能看到什么」时：

1. 运行 `python3 ~/.claude/skills/icost-analysis/scripts/patterns.py [年份1 年份2 ...]`
2. 默认分析 2024+2025+2026 三年，也可指定年份
3. 输出维度：
   - 按星期拆解（周几花最多，工作日 vs 周末倍数）
   - 上旬 vs 下旬对比（月中意志力衰减检测）
   - 晚饭-零食同日绑定率（最危险的消费模式）
   - 晚餐均价走势（逐月/逐年是否涨价）
   - 大额支出后反弹效应（花了大的→吃得更省还是更多）
   - 晚餐月度频次（吃得越来越频繁还是收敛）
4. 解读要点：
   - **星期模式**：周末花费通常是工作日的 2-3 倍——揭示「外部结构 = 刹车」的关系
   - **上下旬**：下旬显著高于上旬意味着月初纪律月中就崩了
   - **绑定率**：>30% 说明零食不是独立冲动，是晚饭决策的附件——切断物理通道比戒零食本身有效
   - **反弹效应**：大件后餐饮不降反升是「破罐破摔」心态——要提醒用户预算不能被单笔大额拖崩

### 手动导入模式

用户上传/在目录放入了新 xlsx 文件，要求立即处理时：

1. **扫描** — `ls ~/records/finance/*.xlsx` 检查是否有新 xlsx
2. **摄入** — `python3 ~/.claude/skills/icost-analysis/scripts/ingest.py --file <路径>`
   - 自动去重追加到年度 csv
   - 自动写入 processed.log
   - 自动删除原始 xlsx
3. **验证** — 检查新数据：`grep "日期" ~/records/finance/2026.csv | tail -N`
4. **出报告** — `python3 ~/.claude/skills/icost-analysis/scripts/report.py`
5. **推送** — 报告直接呈现在对话中（已推送到 QQ）

⚠️ **脚本修复后必做：** 用户说"即刻修复"时，修完先不要报告结果——
   1. 造测试 xlsx → 跑 ingest.py 验证全流程（数据摄入✅ · processed.log写入✅ · 原文件删除✅）
   2. 清理测试数据
   3. `cd ~/.claude/skills && git add/commit/push` — 推送到远程 skills 仓库（"推送"=git push，不是发消息）
   4. 在对话中告知完成
   **用户原话：** "修复完测试一下，测试通过了要推送" — 这里"推送"=git push 到远程仓库
   ❌ 踩坑：上次修完 ingest.py 后，我把"推送"误解为发 QQ 消息 → 被纠正"我是说推送到远程skills仓库"

### 手动查询模式
用户直接问时：
1. 读对应年份 csv
2. 直接出结论：总支出 + 伙食超支 + 零浪费月支出
3. 不列原始数据流水账

## 脚本说明

### ingest.py
- 用法：`python3 scripts/ingest.py --file <xlsx路径>`
- 读取 xlsx 的 `收支账单` sheet
- 仅保留 类型 == "支出" 的记录
- 丢弃图片相关列
- 按日期拆分到年度 csv，去重后追加
- 写入 processed.log（时间戳+文件名）
- 自动删除原始 xlsx

### report.py
- 读年度 csv
- 按预算规则计算各维度
- 每周拆分（从月初开始按自然周划分）
- 生成报告 → print + send_message（推送 QQ）
- QQ 推送失败时打印到终端作为 fallback
