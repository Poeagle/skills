# code-design

将任意代码仓库解构为标准化软件设计文档包，或从零到一输出完整设计规格。

## 关联文件

- `references/architecture-comprehension-via-chain-tracing.md` — 轻量级架构分析方法：沿 forwarder → orchestrator → core 链路阅读分层代码，逐层提取设计决策和边界条件。适用于用户问"解释一下X怎么工作的"或"这个子系统的架构分析"的场景。

- **深入第一**：宁可在单个组件上花更多时间，也不为了"做完了"而草率收尾
- **每一句话都有依据**：设计原理必须追溯到具体的代码证据或业务需求
- **文档即设计**：输出的文档应该精确到可以基于它去修改代码或跨语言重写
- **禁止 Obsidian 双链**：code-design 输出文档中不允许使用 `[[wikilink]]`。组件间的关联引用全部使用纯文本路径（如 `4.组件详情/claude/README` 而不是 `[[4.组件详情/claude/README]]`）。原因是双链会导致 lint 死链误报，且 code-design 文档自成体系，不需要 Obsidian 的图索引能力

## 核心工作流

在开始之前，必须确认用户指定了输出路径。如果用户没有说明输出到哪里，必须追问。不清楚就问，不要假设。

## 模式一：逆向分析（Analyze）

当用户提供代码仓库（本地路径、远程 URL、或已打开的目录）时，自动进入此模式。

### 流程

#### 阶段 1：获取仓库

如果用户提供了远程 URL，先用 `git clone` 克隆到临时目录。如果是本地路径，直接使用。获取后，先记录项目的基本信息：

- 项目名称（从 git remote 或目录名推断）
- 主要语言和技术栈
- 构建系统（Cargo、Maven、npm 等）
- 目录结构概览（只到第一层子目录）

#### 阶段 2：全局浏览

通览整个仓库，识别所有源码子目录作为组件候选。查看项目的入口文件、核心目录结构、关键配置文件。

产出一份组件清单，**按照源码目录结构直接映射，每个源码子目录即为一个组件**：

```
判定组件来源的方法：
1. 以项目主源码目录（如 src/、lib/、packages/ 等）下的第一层子目录作为组件
2. 如果子目录包含多个功能差异明显的模块，可以拆分为子组件
3. 顶级单文件（非 test/config 文件）可作为独立的辅助组件
4. 目录下全是配置/测试/文档的，跳过，不产出空组件
5. 遇到规模巨大的组件（如 200+ 文件），至少产出 README.md 概述
   + 架构概览，不必逐文件拆解到 3.实现步骤.md

禁止按主观重要度筛选组件。源码目录有什么就产什么，全面覆盖，不挑不拣。
```

#### 阶段 3：串行探索组件

对阶段 2 识别出的组件，按重要度从高到低**逐一串行**深入探索。**禁止使用 Agent 工具并行探索**，因为并行会导致信息碎片化、各组件文档之间缺乏连贯性、整体缺少一条主线。

每个组件探索时需回答：

```
该组件的完整代码分析，包括：
- 所有公开的 struct/class、trait/interface 及其职责
- 核心方法的输入输出和执行流程
- 依赖的其他组件
- 异常/错误处理方式
```

#### 阶段 4：产出文档

按以下结构写入用户指定的输出路径。

### 产出目录结构

```
{output_path}/
├── README.md
│   ├── 仓库地址（git remote 或用户提供的 URL）
│   ├── 一句话说清这个项目是什么（给小白看的，不要术语堆砌）
│   ├── 用户旅程：用一段话描述"一个用户从头到尾使用这个仓库的完整流程"
│   ├── 小白浏览步骤：按什么顺序读本文档，每篇文档是干什么的
│   ├── 技术栈一览
│   ├── 整体架构概述（3-5句话）
│   └── 如何基于本文档修改项目（入口说明）
│
├── 1.架构概览.md
│   ├── 项目简介与目标
│   ├── 技术栈与约束（语言、框架、构建工具、运行环境）
│   ├── 系统上下文（和外部系统的关系、依赖）
│   ├── 分层架构图 + 模块依赖关系
│   │       （用 ASCII 框图呈现）
│   ├── 数据模型概览（如果项目使用数据库，说明数据库类型和技术选型）
│   └── ER 关系图（如果项目使用数据库，用 ASCII 表格画出核心表之间的关联）
│
├── 2.通信图.md
│   ├── 核心流程时序图（启动流程、一次对话轮次等）
│   ├── 模块间调用关系
│   └── 关键数据流（数据从哪里产生、经过哪些处理、最终到哪里）
│
├── 3.跨切面概念.md
│   ├── 配置系统
│   ├── 错误处理模式
│   ├── 日志/遥测
│   ├── 安全/权限
│   └── 测试策略
│
├── 4.组件详情/
│   ├── {组件名称}/
│   │   ├── README.md         ← 组件一句话职责 + 输入输出 + 依赖关系
│   │   ├── 1.设计原理.md      ← 为什么这么设计、解决了什么痛点
│   │   ├── 2.架构.md          ← 内部结构（struct/class、trait/interface、子模块关系图）
│   │   └── 3.实现步骤.md      ← 核心方法的完整流程拆解（带代码片段和逐段解读）
│   └── ...
│
│   ⚠️ 硬性要求：每个组件目录下的 4 个文件（README.md、1.设计原理.md、
│      2.架构.md、3.实现步骤.md）必须全部写完，少一个就是不合格，
│      必须重写。禁止创建空目录。一个组件没写完绝对禁止跳到下一个。
│

    ├── 部署架构

### 组件详情文件规范

**README.md** — 模块地图（给 agent 看：有哪些文件、每个干什么）
```markdown
## 职责
用通俗语言说明这个组件做什么（1-2句，不准堆砌术语）。

## 子模块清单
列出该组件下的**所有**非测试源文件，按功能分组：

| 文件 | 职责 |
|------|------|
| `xxx.ts` | 做什么，暴露了哪些关键函数/类 |
| `yyy.ts` | 做什么 |
| ... | ... |

（必须列出全部源文件，不能只挑代表。如果文件超过 20 个，按子目录分组列表。）

## 输入
该组件接收什么数据/事件/配置。

## 输出
该组件产出什么数据/结果/副作用。

## 依赖
- `[组件名/模块路径]` — 依赖的用途

## 被依赖
- `[组件名]` — 在什么场景下依赖本组件
```

**1.设计原理.md** — 设计约束（给 agent 看：为什么这么设计，改了会破坏什么）
```markdown
## 问题背景
这个组件在解决什么具体问题？描述痛点，不要复述需求。

## 设计思路
为什么选择这个方案？至少列出 3 个对比方案，用表格对比：

| 方案 | 优点 | 缺点 | 选择理由 |
|------|------|------|---------|
| 当前方案 | ... | ... | 为什么选 |
| 方案 B | ... | ... | 为什么不选 |
| 方案 C | ... | ... | 为什么不选 |

## 架构决策记录
列出关键的设计取舍：

| 决策项 | 选择 | 替代方案 | 理由 |
|--------|------|---------|------|
| ... | ... | ... | ... |

## 边界范围
明确这个组件不做什么、不负责什么（防止 agent 越界修改）。
```

**2.架构.md** — 代码地图（给 agent 看：代码在哪、怎么组织的）
```markdown
## 文件清单
按功能分组列出该组件下的**所有**非测试源文件：

**分组名**（如"核心逻辑"）：
- `xxx.ts` — 职责
- `yyy.ts` — 职责

**分组名**（如"工具函数"）：
- ...

## 核心接口与类型
列出所有 `export` 的 interface / type / class，标注定义文件和用途：

| 类型 | 定义文件 | 用途 |
|------|---------|------|
| `InterfaceName` | `xxx.ts` | 做什么 |
| `TypeName` | `yyy.ts` | 描述 |

## 模块关系图
用 ASCII 框图展示模块内部结构、文件间依赖关系。

## 核心数据流
关键路径的数据流向，用步骤列表或 ASCII 图描述：
1. 从哪来 → 经过什么处理 → 到哪去
2. ...
```

**3.实现步骤.md** — 手术指南（给 agent 看：怎么改、影响哪里）
```markdown
## 总体流程
先给出高层步骤概览，让读者看到全景再深入细节：
1. 第一步做什么
2. 第二步做什么
3. 第三步做什么
（3-5 步，每步一句话）

## 详细拆解
### 步骤1：[步骤名称]
代码片段（从 Read 工具读取的实际源码，注明文件路径）：
```typescript
// src/xxx/yyy.ts:42-48
function example() {
  // ...
}
```
逐段解读：这段代码在做什么、为什么这么做、关键变量的状态变化。

### 步骤2：[步骤名称]
...

## 关键分支
异常路径、错误处理、边界条件。列出至少两个非正常路径的处理方式。

## 调用关系
- **被谁调用**：这个方法/模块被哪些模块在什么场景下调用
- **调用了谁**：这个方法/模块内部调用了哪些其他模块
（给 agent 看的影响面分析，修改时知道要改哪些地方）
```

## 模式二：正向蓝图（Blueprint）

当用户描述一个想法、需求、或者想从零搭建项目时，自动进入此模式。

### 流程

#### 阶段 1：需求捕获

当用户给出的信息不完整时，用提问引导用户澄清以下内容：

```
1. 这个项目要解决什么问题？（目标）
2. 谁会用它？（用户）
3. 核心功能有哪些？（功能清单）
4. 有什么技术约束？（语言偏好、平台、性能要求）
5. 有没有时间或资源限制？
6. 有没有参考资料或类似产品？
```

不要一次性抛出所有问题。像产品经理一样，在一次对话中逐步追问。问完一个，理解清楚，再问下一个。

#### 阶段 2：产出 PRD

将需求整理为结构化的《需求规格说明书》，内容包括：

```
1. 产品目标（1-2句话）
2. 用户故事（as a...I want...so that...）
3. 功能清单（必须做 / 应该做 / 可选的）
4. 非功能需求（性能、安全、可扩展性）
5. 验收标准
6. 边界范围（明确不做什么）
```

输出后请用户确认是否满足预期，不满足则修改。

#### 阶段 3：产出技术方案

在用户确认 PRD 后，产出技术方案：

```
1. 技术选型（语言、框架、存储、基础设施）——附选择理由
2. 系统上下文图（系统和外部实体的交互）
3. 分层架构
4. 数据模型（核心实体关系）
5. 模块划分
```

同样需要用户确认。

#### 阶段 4：产出接口规格

在技术方案确认后，产出接口规格：

```
1. 模块间接口定义
2. 外部 API 定义（路径、方法、请求/响应格式）
3. 事件/消息格式
4. 错误码定义
```

同样需要用户确认。

#### 阶段 5：产出组件详情

参考"逆向分析"模式中的组件详情格式，写出每个组件的设计。如果组件较多，按重要度排序，先写核心组件，再写辅助组件。

### 产出目录结构

```
{output_path}/
├── README.md
│   ├── 项目简介
│   └── 文件导航说明
│
├── 1.需求规格（PRD）.md
│   ├── 产品目标
│   ├── 用户故事
│   ├── 功能清单
│   ├── 非功能需求
│   └── 验收标准
│
├── 2.技术方案.md
│   ├── 技术选型与依据
│   ├── 系统上下文
│   ├── 分层架构
│   └── 数据模型
│
├── 3.接口规格.md
│   ├── 模块间接口
│   ├── 外部 API 定义
│   └── 事件/消息格式
│
└── 4.组件详情/
    └── (格式同逆向分析模式)
```

## 质量要求

分析代码时：
- **禁止创建空目录**——每个组件目录下的 4 个文件（README.md、1.设计原理.md、2.架构.md、3.实现步骤.md）必须全部写完。少一个就是不合格，必须重写
- **组件必须从源码目录结构识别**——源码目录有多少个子目录，就应该产出多少组件目录。不能漏、不能挑。每个子目录的 README.md 必须写清楚该模块的职责边界。
- **必须完整阅读每个组件的源码**——不能只看文件列表就下结论。每个组件的设计原理必须有具体的代码行号作为引用证据
- **每一段分析都要有代码依据**——说"这个组件用了策略模式"就要指出对应的 trait 和实现
- **关键方法必须拆解到行级别**——核心方法（如 `run_turn()`、`execute_agent()`）的执行流程必须逐段解释，附代码片段和逐行解读
- **不要遗漏公共 API**——所有公开的类型、trait、函数都应被记录
- **数据库设计必须完整**——如果项目使用数据库（SQL/NoSQL），必须在 `1.架构概览.md` 中输出完整的表结构设计和 ER 关系图（ASCII 表格+关系连线），包括每个表的字段名、类型、约束、索引和外键关系

### 🔴 源码探索硬标准（违反即不合格）

以下为硬性要求，不满足则文件不合格，必须重写：

**1. 源码阅读要求：写之前必须读源码**

写一个组件的任何文件之前，必须先用 Read 工具读取该组件的实际源文件：

| 组件规模 | 要求 |
|---------|------|
| 1-10 个源文件 | **全部读取**（排除 `.test.ts`） |
| 11-50 个源文件 | 读取 **不少于 80%** 的源文件 |
| 50+ 个源文件 | 读取主入口、类型定义、核心实现（不少于 15 个关键文件） |

判断标准：看工具调用记录，如果 Read 工具没有读取足够数量的该组件源文件就开始写，视为欺诈，已写的文件全部删除重做。

**2. 代码引用要求：每个文档必须有实际源码依据**

- `1.设计原理.md` 中每个设计决策必须指向具体的文件名或函数名
- `3.实现步骤.md` 中每个代码片段必须来自 Read 工具的实际输出，禁止"凭记忆"写代码
- 引用格式：在代码片段前注明 `// src/xxx/yyy.ts`，行号可选但鼓励标注
- 如果 Read 的输出中没有该段代码，说明没有读那个文件，必须补读后再写

**3. 内容要求：覆盖组件的全部功能**

- README.md 必须列出该组件的**所有**子模块/文件组（不是挑几个代表）
- `1.设计原理.md` 必须覆盖该组件解决的**所有**主要问题（不是只写最明显的一个）
- `2.架构.md` 必须包含**完整的**文件清单或模块树
- `3.实现步骤.md` 必须覆盖**至少两个**不同的核心流程（如读和写、创建和删除）

**4. 禁止行为：**
- 禁止未读源码就下结论
- 禁止跨组件复制粘贴改几个字交差
- 禁止"等"、"等等"、"..."等模糊省略——列全或者不列
- 禁止"后续补充"、"待完善"、"TODO"等占位文字
- 禁止写自己没通过 Read 确认过的代码片段

**5. 自检流程：**
- 每个组件写完后，检查 Read 工具调用记录中是否确实读取了该组件的源码
- 如果某个文件的内容无法追溯到 Read 操作，删除重写

组件详情产出顺序规范：
1. 每次只处理**一个**组件
2. 处理步骤：探索源码 → 写 README.md → 写 1.设计原理.md → 写 2.架构.md → 写 3.实现步骤.md
3. **禁止批量操作**：禁止用 Bash 脚本批量创建目录、批量写文件、批量 Read 源码。每次 Read 只读一个文件，每次 Write 只写一个文件
4. **禁止"先建所有目录"**：不要提前建好所有组件的空目录。目录随着组件处理逐个创建
5. **一个组件没写完（4 个文件全部完成且内容完整），绝对禁止跳到下一个组件**
6. **禁止并行。** 串行是保证文档连贯性和主线清晰的前提

写文档时：
- **不要写模糊的概括**——"该项目采用模块化设计"这种话没有信息量，要说清楚"分为 xxx 模块，分别负责 yyy"
- **禁止使用 Mermaid 图表**——Mermaid 在 Obsidian 中会撑破屏幕且渲染卡死。所有图表必须使用 ASCII 格式，在 ` ``` ` 代码块中用纯文本绘制序列图、架构图、流程图。确保每行不超过 80 字符，在终端中可直接阅读。
- **每个文件必须完整**——一个组件的 4 个文件全部写完再进入下一个组件，不要留"待补充"

### 🔴 绝对铁律：禁止任何形式的双链

**这条规则是所有规则中优先级最高的，违反将导致文档不合格。**

**规则内容**：
- 产出物是独立的设计文档目录，不是 wiki 页面
- 绝对禁止在文档中出现 `[[xxx]]` 格式的双链——它们不会指向任何文件，产生死链
- 无论是 README.md、架构概览、通信图，还是组件详情中的任何文件，一个 `[[` 都不能出现
- 如需引用其他组件，使用 Markdown 链接如 `[Engine](组件详情/Engine/)` 或直接用文字表示
- **在 spawn 子 agent 或写 prompt 给任何子任务时，必须显式传递这条规则**

**惩罚机制**：如果最终产出物中出现任何 `[[` 格式文本，必须立即删除并重写该文件。

### 📖 主线与用户友好

- README.md 必须包含"用户旅程"和"小白浏览步骤"
- 每个组件在说明文档时必须先有一句"这个组件是干什么的"，用通俗语言表达，不准用技术术语堆砌
- 所有文档写完后必须从头读一遍，确保一个完全不懂这个项目的人也能理解

对于用户来说：
- **不猜测用户意图**——如果用户的需求模糊，必须追问清楚再开始
- **不跳过确认步骤**——特别是正向蓝图模式的阶段 2/3/4，每阶段产出都需要用户确认
- **不接受默认路径**——用户必须明确说出输出位置

## 硬约束

- 产出物必须写入用户显式指定的路径。未指定时持续追问，不假设。
- 逆向分析时，如果项目没有按功能划分模块，向用户说明原因并停止。
- 正向蓝图时，每个阶段都需要用户确认后再进入下一阶段。
