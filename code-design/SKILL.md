# code-design

将任意代码仓库解构为标准化软件设计文档包，或从零到一输出完整设计规格。

## 核心原则

- **深入第一**：宁可在单个组件上花更多时间，也不为了"做完了"而草率收尾
- **每一句话都有依据**：设计原理必须追溯到具体的代码证据或业务需求
- **文档即设计**：输出的文档应该精确到可以基于它去修改代码或跨语言重写

## 核心工作流

在开始之前，必须确认用户指定了输出路径。如果用户没有说明输出到哪里，必须追问。不清楚就问，不要假设。

---

## 模式一：逆向分析（Analyze）

当用户提供代码仓库（本地路径、远程 URL、或已打开的目录）时，自动进入此模式。

### 流程

#### 阶段 1：获取仓库

如果用户提供了远程 URL，先用 `git clone` 克隆到临时目录。如果是本地路径，直接使用。获取后，先记录项目的基本信息：

- 项目名称（从 git remote 或目录名推断）
- 主要语言和技术栈
- 构建系统（Cargo、Maven、npm 等）
- 目录结构概览（只到第一层子目录）

#### 阶段 2：全局浏览

通览整个仓库，识别模块/组件划分。查看项目的入口文件、核心目录结构、关键配置文件。

产出一份组件清单，按重要度排序：

```
判定组件重要度的标准（从高到低）：
1. 入口/启动类 — 项目从哪里开始运行
2. 核心抽象 — 项目中最核心的 trait/interface
3. 主循环/调度器 — 驱动系统运行的循环
4. 数据模型 — 核心数据结构
5. 外部接口 — 和其他系统通信的边界
6. 工具/辅助模块 — 支撑功能

如果项目没有按功能划分模块（组件），则不值得产出文档，向用户说明原因。
```

#### 阶段 3：并行探索组件

对阶段 2 识别出的组件，按重要度从高到低依次深入探索。可以使用 Agent 工具启动子 agent 并行探索不同的组件。

每个组件探索时需回答：

```
该组件的完整代码分析，包括：
- 所有公开的 struct/class、trait/interface 及其职责
- 核心方法的输入输出和执行流程
- 依赖的其他组件
- 异常/错误处理方式
```

#### 阶段 4：产出文档

按以下结构写入用户指定的输出路径。

### 产出目录结构

```
{output_path}/
├── README.md
│   ├── 仓库地址（git remote 或用户提供的 URL）
│   ├── 项目一句话介绍
│   ├── 技术栈一览
│   ├── 整体架构概述（3-5句话）
│   └── 如何基于本文档修改项目（入口说明）
│
├── 1.架构概览.md
│   ├── 项目简介与目标
│   ├── 技术栈与约束（语言、框架、构建工具、运行环境）
│   ├── 系统上下文（和外部系统的关系、依赖）
│   ├── 分层架构图 + 模块依赖关系
│   │       （用 mermaid 或 excalidraw 图表呈现）
│   ├── 数据模型概览（如果项目使用数据库，说明数据库类型和技术选型）
│   └── ER 关系图（如果项目使用数据库，用 mermaid 画出核心表之间的关联）
│
├── 2.通信图.md
│   ├── 核心流程时序图（启动流程、一次对话轮次等）
│   ├── 模块间调用关系
│   └── 关键数据流（数据从哪里产生、经过哪些处理、最终到哪里）
│
├── 3.跨切面概念.md
│   ├── 配置系统
│   ├── 错误处理模式
│   ├── 日志/遥测
│   ├── 安全/权限
│   └── 测试策略
│
├── 4.组件详情/
│   ├── {组件名称}/
│   │   ├── README.md         ← 组件一句话职责 + 输入输出 + 依赖关系
│   │   ├── 1.设计原理.md      ← 为什么这么设计、解决了什么痛点
│   │   ├── 2.架构.md          ← 内部结构（struct/class、trait/interface、子模块关系图）
│   │   └── 3.实现步骤.md      ← 核心方法的完整流程拆解（带代码片段和逐段解读）
│   └── ...
│
└── 5.部署视图.md（可选）
    ├── 部署架构
    ├── 环境要求
    └── 启动/运行方式
```

### 组件详情文件规范

**README.md**：
```markdown
## 职责
一句话说明这个组件做什么。

## 输入
该组件接收什么数据/事件。

## 输出
该组件产出什么数据/结果。

## 依赖（Dependencies）
- [组件A] — 用途
- [组件B] — 用途

## 被依赖（Dependents）
- [组件X] — 在什么场景下依赖本组件
```

**1.设计原理.md**：
```markdown
## 问题背景
这个组件解决了什么痛点？

## 设计思路
为什么选择这个方案？和其他可能的方案对比。

## 对应的设计模式
可以用什么经典设计模式来描述？（trait 对象→策略模式、观察者等）

## 架构决策记录
关键的设计选择和取舍。
```

**2.架构.md**：
```markdown
## 核心结构体/类
列出所有公开的类型及其职责。

## 核心接口/Trait
列出所有抽象接口及其用途。

## 内部架构图
用 mermaid 或 excalidraw 展示内部结构关系。

## 模块划分
内部的子模块拆分及其边界。
```

**3.实现步骤.md**：
```markdown
## 核心流程
关键方法的执行步骤拆解，每一步配上：
- 代码片段（截取最关键的部分）
- 逐段解读（这段代码在做什么、为什么这么做）
- 状态变化（如果有状态改变，记录前后的状态）

## 关键分支
异常路径、错误处理、边界条件。

## 生命周期
组件从创建到销毁的完整过程。
```

## 模式二：正向蓝图（Blueprint）

当用户描述一个想法、需求、或者想从零搭建项目时，自动进入此模式。

### 流程

#### 阶段 1：需求捕获

当用户给出的信息不完整时，用提问引导用户澄清以下内容：

```
1. 这个项目要解决什么问题？（目标）
2. 谁会用它？（用户）
3. 核心功能有哪些？（功能清单）
4. 有什么技术约束？（语言偏好、平台、性能要求）
5. 有没有时间或资源限制？
6. 有没有参考资料或类似产品？
```

不要一次性抛出所有问题。像产品经理一样，在一次对话中逐步追问。问完一个，理解清楚，再问下一个。

#### 阶段 2：产出 PRD

将需求整理为结构化的《需求规格说明书》，内容包括：

```
1. 产品目标（1-2句话）
2. 用户故事（as a...I want...so that...）
3. 功能清单（必须做 / 应该做 / 可选的）
4. 非功能需求（性能、安全、可扩展性）
5. 验收标准
6. 边界范围（明确不做什么）
```

输出后请用户确认是否满足预期，不满足则修改。

#### 阶段 3：产出技术方案

在用户确认 PRD 后，产出技术方案：

```
1. 技术选型（语言、框架、存储、基础设施）——附选择理由
2. 系统上下文图（系统和外部实体的交互）
3. 分层架构
4. 数据模型（核心实体关系）
5. 模块划分
```

同样需要用户确认。

#### 阶段 4：产出接口规格

在技术方案确认后，产出接口规格：

```
1. 模块间接口定义
2. 外部 API 定义（路径、方法、请求/响应格式）
3. 事件/消息格式
4. 错误码定义
```

同样需要用户确认。

#### 阶段 5：产出组件详情

参考"逆向分析"模式中的组件详情格式，写出每个组件的设计。如果组件较多，按重要度排序，先写核心组件，再写辅助组件。

### 产出目录结构

```
{output_path}/
├── README.md
│   ├── 项目简介
│   └── 文件导航说明
│
├── 1.需求规格（PRD）.md
│   ├── 产品目标
│   ├── 用户故事
│   ├── 功能清单
│   ├── 非功能需求
│   └── 验收标准
│
├── 2.技术方案.md
│   ├── 技术选型与依据
│   ├── 系统上下文
│   ├── 分层架构
│   └── 数据模型
│
├── 3.接口规格.md
│   ├── 模块间接口
│   ├── 外部 API 定义
│   └── 事件/消息格式
│
└── 4.组件详情/
    └── (格式同逆向分析模式)
```

## 质量要求

分析代码时：
- **不要快速浏览放弃**——每个组件的代码必须完整阅读，不能只看文件列表就下结论
- **每一段分析都要有代码依据**——说"这个组件用了策略模式"就要指出对应的 trait 和实现
- **关键方法必须拆解到行级别**——run_turn() 这种核心方法的每一段都要解释
- **不要遗漏公共 API**——所有 pub 函数和 pub struct 都应被记录
- **数据库设计必须完整**——如果项目使用数据库（SQL/NoSQL），必须在 `1.架构概览.md` 中输出完整的表结构设计和 ER 关系图（mermaid），包括每个表的字段名、类型、约束、索引和外键关系

写文档时：
- **不要写模糊的概括**——"该项目采用模块化设计"这种话没有信息量，要说清楚"分为 xxx 模块，分别负责 yyy"
- **图表必须有意义**——mermaid/excalidraw 图要展示真实的结构关系，不是画个框写个名字
- **每个文件必须完整**——一个组件的 4 个文件全部写完再进入下一个组件，不要留"待补充"
- **禁止使用 Obsidian 双链**——产出物是独立的设计文档目录，不是 wiki 页面。`[[runtime]]`、`[[api]]` 等双链不会指向任何现有文件，会造成死链。如需引用其他组件，使用 Markdown 链接如 `[runtime](组件详情/runtime/)` 或普通文字

对于用户来说：
- **不猜测用户意图**——如果用户的需求模糊，必须追问清楚再开始
- **不跳过确认步骤**——特别是正向蓝图模式的阶段 2/3/4，每阶段产出都需要用户确认
- **不接受默认路径**——用户必须明确说出输出位置

## 硬约束

- 产出物必须写入用户显式指定的路径。未指定时持续追问，不假设。
- 逆向分析时，如果项目没有按功能划分模块，向用户说明原因并停止。
- 正向蓝图时，每个阶段都需要用户确认后再进入下一阶段。
