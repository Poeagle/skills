# lint 技能

## 扫描
运行本技能目录下 scripts/lint.py 脚本，读取其输出的 JSON 作为健康报告。

脚本输出包含以下增强字段：
- `pending_file_list`：待处理文件的完整路径列表（不再只是数字）
- 死链报告中的 `wikilink` / `source_page` / `target_name`（不再只有长路径）
- code-design 缺陷中的 `[ERROR]` / `[WARN]` 前缀（区分严重级别）
- `placeholder_count`：子标题下包含占位符（待补充/TODO）的数量

## 报告格式
按以下模板呈现。从 JSON 结果中提取数据时：

- ​**汇总行**：首行输出 "N 页，M 条链接，X 死链，Y 孤儿" 摘要
- 死链路径用 `source_page → [[target_name]]` 格式（来自脚本的 wikilink/source_page/target_name 字段），而非原始文件路径
- 待处理文件列表用 `pending_file_list` 字段，而非仅使用 `pending_files` 数字

### ✅ 绿灯项
- **知识库概览**：N 页，M 条链接，X 死链，Y 孤儿，Z 冲突
- [运行正常的其他项目]

### ⚠️ 黄灯项
- **发现 N 个孤儿页面**：[列表] - 建议添加关联或分类
- **发现 N 个未同步索引**：[列表] - 文件存在但未在 index.md 注册
- **待处理文件**：N 个（根据 `pending_file_list` 列出具体路径）
- **code-design 警告**：[WARN] 类缺陷（缺失子标题、子标题内容为空或含占位符）

### ❌ 红灯项
- **发现 N 个死链**：`source_page → [[target_name]]`
- **存在 N 个未解决的知识冲突**：[页面名称]
- **code-design 错误**：[ERROR] 类缺陷（缺失文件）

### 🛠️ 下一步行动
1. 是否需要自动修复未同步索引？
2. 是否需要针对知识冲突进行重新推演？
3. 是否需要补充包含占位符的 code-design 组件内容？

## code-design 专项检查

如果 `wiki/code-design/` 目录存在，lint.py 会自动执行 code-design 完整性检查，输出结果在 `code_design` 字段中。每个仓库包含：

| 字段 | 含义 |
|------|------|
| `repo` | 仓库名 |
| `total_components` | 组件总数 |
| `passed` | 合格组件数 |
| `failed` | 不合格组件数 |
| `defects` | 缺陷列表（含组件名、缺失文件、缺失子标题） |

**检查规则**（脚本自动执行）：
- **文件完整性**：[ERROR] 级——每个组件必须有 README.md、1.设计原理.md、2.架构.md、3.实现步骤.md
- **子标题完整性**：[WARN] 级——每个文件必须包含规定子标题（具体规则见脚本中的 `REQUIRED_HEADINGS`）
- **内容质量**：[WARN] 级——子标题下内容不得为空或仅含"待补充"/"TODO"等占位符

**缺陷级别的含义**：
- `[ERROR]` 缺失文件：必须修复，否则组件不完整
- `[WARN]` 缺失子标题或占位符：建议修复，但允许在用户确认的情况下暂时保留

### 修复流程（用户确认后执行）

**核心原则：一个组件一个组件来，严禁批量处理。按优先级从高到低处理：**

0. **修复前确认**：向用户展示本次 code-design 检查摘要（passed/failed，ERROR/WARN 数量），等待用户确认是否开始修复
1. **优先修复 [ERROR] 类缺陷**（缺失文件）：这些是硬性缺陷
2. **再修复 [WARN] 类缺陷**（缺失子标题、占位符）：按组件从核心到外围逐个修复
3. **补读源码**：遵循 `code-design` skill 的源码探索硬标准
4. **重新生成**：严格按照 `code-design` skill 的文档模板和内容要求重新生成
5. **自检**：确认所有子标题和内容完整，无占位符残留
6. **进入下一个**：当前组件完全修复后，再进入下一个

---

## 修复（用户确认后执行）
- **未同步索引** → 向 wiki/index.md 对应分类下追加条目
- **死链** → 逐条判断：拼写错误则修正，页面删除则评估是否重建
- **孤儿页面** → 添加 ## 关联连接 或归入分类
- **修复完成** → 在 wiki/log.md 追加日志

## 硬约束
- 仅读扫描，生成报告前不修改任何文件
- 报告后等用户确认再执行修复
- 修复后在 log.md 追加操作日志
