# lint 技能

## 扫描
运行本技能目录下 scripts/lint.py 脚本，读取其输出的 JSON 作为健康报告。

## 报告格式
按以下模板呈现：

### ✅ 绿灯项
- [运行良好的项目]

### ⚠️ 黄灯项
- **发现 N 个孤儿页面**：[列表] - 建议添加关联或分类
- **发现 N 个未同步索引**：[列表] - 文件存在但未在 index.md 注册

### ❌ 红灯项
- **发现 N 个死链**：[来源页面] → [[不存在的目标页面]]
- **存在 N 个未解决的知识冲突**：[页面名称]

### 🛠️ 下一步行动
1. 是否需要自动修复未同步索引？
2. 是否需要针对知识冲突进行重新推演？

## code-design 专项检查

如果 `wiki/code-design/` 目录存在，在上述通用检查**之后**，必须执行以下专项检查。

### 检查步骤

#### 步骤 1：遍历所有仓库
扫描 `wiki/code-design/` 下的每个仓库目录（如 `wiki/code-design/gbrain/`、`wiki/code-design/openclaw/` 等）。

#### 步骤 2：检查组件完整性
对每个仓库目录，遍历 `4.组件详情/` 下的每个子目录，逐项检查：

| 检查项 | 规则 | 判定为缺陷的条件 |
|--------|------|-----------------|
| 文件完整性 | 每个组件目录必须有 4 个文件 | 缺少 `README.md`、`1.设计原理.md`、`2.架构.md`、`3.实现步骤.md` 中任意一个 |
| 子标题完整性 | 每个文件必须包含规定的子标题 | 对照 `code-design` skill 中的文档模板逐项检查 |

**文件级子标题检查标准**（参照 code-design SKILL.md 中的文档模板）：

1. **README.md** 必须包含：`## 职责`、`## 子模块清单`（或 `## 文件清单`）、`## 输入`、`## 输出`、`## 依赖`、`## 被依赖`
2. **1.设计原理.md** 必须包含：`## 问题背景`、`## 设计思路`（或 `## 架构决策记录` / `## 架构决策`）、`## 边界范围`
3. **2.架构.md** 必须包含：`## 文件清单`、`## 核心接口与类型`、`## 模块关系图`、`## 核心数据流`
4. **3.实现步骤.md** 必须包含：`## 总体流程`、`## 详细拆解`（至少 2 个步骤）、`## 关键分支`（至少 2 个）、`## 调用关系`

> **判定原则**：一个组件只要有一个文件缺失，就视为完整文件缺失；一个文件只要有一个规定子标题缺失，就视为子标题缺失。一个组件只要存在两类缺陷中的任意一类，即判定为**不合格组件**。

#### 步骤 3：报告缺陷
在通用健康报告之后，追加 code-design 专项报告：

```markdown
### 📐 code-design 完整性检查

**仓库：[仓库名]**
- **合格组件**：N 个（列表）
- **不合格组件**：N 个（列表 + 具体缺陷说明）
  - [组件名]：缺失 [具体文件名]，缺失子标题 [具体标题名]
  - [组件名]：缺失子标题 [具体标题名]
```

### 修复流程（用户确认后执行）

**核心原则：一个组件一个组件来，严禁批量处理。**

1. **选择不合格组件**：从最核心（或最严重）的组件开始，每次只处理一个
2. **补读源码**：如果该组件的文档需要重新生成，先重新读取该组件的实际源码（遵循 code-design 的源码探索硬标准）
3. **重新生成**：严格按照 `code-design` skill 的文档模板和内容要求重新生成全部 4 个文件
4. **自检**：重新检查修复后的组件，确认所有子标题和内容完整
5. **进入下一个**：当前组件完全修复后，再进入下一个组件
6. **绝不允许**：同时处理多个组件、批量写入文件、或跳过源码阅读直接重写

> 如果所有组件均合格，在报告中标注"code-design 完整性检查通过 ✅"。

## 修复（用户确认后执行）
- **未同步索引** → 向 wiki/index.md 对应分类下追加条目
- **死链** → 逐条判断：拼写错误则修正，页面删除则评估是否重建
- **孤儿页面** → 添加 ## 关联连接 或归入分类
- **修复完成** → 在 wiki/log.md 追加日志

## 硬约束
- 仅读扫描，生成报告前不修改任何文件
- 报告后等用户确认再执行修复
- 修复后在 log.md 追加操作日志
