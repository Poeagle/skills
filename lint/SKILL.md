# lint 技能

## 扫描
运行本技能目录下 scripts/lint.py 脚本，读取其输出的 JSON 作为健康报告。

## 报告格式
按以下模板呈现：

### ✅ 绿灯项
- [运行良好的项目]

### ⚠️ 黄灯项
- **发现 N 个孤儿页面**：[列表] - 建议添加关联或分类
- **发现 N 个未同步索引**：[列表] - 文件存在但未在 index.md 注册

### ❌ 红灯项
- **发现 N 个死链**：[来源页面] → [[不存在的目标页面]]
- **存在 N 个未解决的知识冲突**：[页面名称]

### 🛠️ 下一步行动
1. 是否需要自动修复未同步索引？
2. 是否需要针对知识冲突进行重新推演？

## code-design 专项检查

如果 `wiki/code-design/` 目录存在，lint.py 会自动执行 code-design 完整性检查，输出结果在 `code_design` 字段中。每个仓库包含：

| 字段 | 含义 |
|------|------|
| `repo` | 仓库名 |
| `total_components` | 组件总数 |
| `passed` | 合格组件数 |
| `failed` | 不合格组件数 |
| `defects` | 缺陷列表（含组件名、缺失文件、缺失子标题） |

**检查规则**（脚本自动执行）：
- **文件完整性**：每个组件必须有 README.md、1.设计原理.md、2.架构.md、3.实现步骤.md
- **子标题完整性**：每个文件必须包含规定子标题（具体规则见脚本中的 `REQUIRED_HEADINGS`）

### 修复流程（用户确认后执行）

**核心原则：一个组件一个组件来，严禁批量处理。**

1. **选择不合格组件**：从最核心的组件开始，每次只处理一个
2. **补读源码**：遵循 `code-design` skill 的源码探索硬标准
3. **重新生成**：严格按照 `code-design` skill 的文档模板和内容要求重新生成全部 4 个文件
4. **自检**：确认所有子标题和内容完整
5. **进入下一个**：当前组件完全修复后，再进入下一个

## 修复（用户确认后执行）
- **未同步索引** → 向 wiki/index.md 对应分类下追加条目
- **死链** → 逐条判断：拼写错误则修正，页面删除则评估是否重建
- **孤儿页面** → 添加 ## 关联连接 或归入分类
- **修复完成** → 在 wiki/log.md 追加日志

## 硬约束
- 仅读扫描，生成报告前不修改任何文件
- 报告后等用户确认再执行修复
- 修复后在 log.md 追加操作日志
