# query 技能

## 核心目标
将用户的提问转化为对本地 Wiki 的深度检索。提取相关页面信息，综合出带有明确引用来源的双链回答。当回答具有高价值时，主动将其固化为知识库的一部分。

## 触发场景（任一即触发，必须先搜 wiki 再回答）

**显式触发：**
- 用户输入 `/query <问题>`
- 用户提及 wiki、知识库、笔记、记录等关键词

**隐式触发（最容易遗漏，必须主动识别）：**
- 用户问关于**人物、概念、理论、哲学、方法论**的知识性问题
- 用户问 **A 和 B 的对比/异同**（如"X 和 Y 有什么共同点"）
- 用户问关于**某人/某流派的观点、思想、生平**
- 任何你**直觉上觉得 wiki 可能有答案**的问题

> ⚠️ **核心规则：如果 wiki/index.md 里有相关内容，你就不能凭记忆回答。**
> 用户建 wiki 的目的就是让你用它。凭训练数据回答 = 浪费了知识库 = 用户会不满。

## 降级策略
如果问题属于纯通用知识（如"太阳系有几颗行星"），**且 wiki/index.md 中确实无相关内容**：
> 本地知识库中未找到相关内容，以下为通用知识回答：[直接回答]

但降级是**例外**，不是默认。先搜再判断，不要先判断要不要搜。

## 检索与综合流水线

### 步骤 1：查阅全局索引
**永远的第一步**：读取 `wiki/index.md`

在 index.md 中定位与问题相关的：
- Entities（实体）
- Concepts（概念）
- Sources（摘要）
- Syntheses（综合）

### 步骤 2：代码实现类问题的特殊检索路径

如果用户的提问涉及**代码实现、架构设计、组件内部逻辑**（如"某功能是怎么实现的"、"某模块的架构是什么"、"如何修改某处代码"），优先执行以下检索路径：

#### 2a. 问题分类：先判断需要什么级别的信息

在开始检索前，先判断用户的问题类型，这决定了是否必须追溯源码：

| 问题类型 | 特征 | 检索深度 |
|---------|------|---------|
| **概览类** | "整体架构是什么"、"有哪些组件"、"怎么部署" | 设计文档即可 |
| **机制类** | "到底是A还是B"、"到底是怎么工作的"、"内部路径是什么" | **必须追溯源码**。设计文档描述意图，不描述实现细节——文档可以说"混合搜索"，但不说 SQL 里的 `chunk_source` 过滤逻辑和 RRF 融合系数 |
| **修改类** | "怎么加一个X功能"、"怎么修Y问题" | 设计文档 + 源码追溯 |

> 坑：**机制类问题最常踩的坑就是用设计文档代替源码。** 当你需要回答"到底是X还是Y"、"内部路径的具体分支是什么"这类问题时，设计文档永远不够——它写的是一篇说明文，不是代码执行路径。必须追溯到实际的 handler/SQL/条件分支。

#### 2b. 查找 code-design 文档

检查 `wiki/code-design/` 下是否有对应仓库的设计文档。例如，关于某个仓库的实现问题，先定位到 `wiki/code-design/{仓库名}/`。

#### 2c. 阅读项目级架构文档

在设计文档目录下，先按顺序阅读项目级文档以建立整体理解，再深入组件细节：
   - `README.md` — 项目概述、技术栈、组件清单、部署视图
   - `1.架构概览.md` — 系统定位、分层架构图、核心设计决策
   - `2.通信图.md` — 请求处理流程、模块间时序图、数据流
   - `3.跨切面概念.md` — 配置系统、错误处理、安全、测试策略等横切关注点
   - （如果存在）`5.部署视图.md` — 生产部署方案
   > 坑：不要跳过项目级架构文档直接读组件细节。没有整体理解时，组件的设计决策（为何这样实现）无法被解释清楚。

#### 2d. 深度阅读组件文档

建立整体理解后，根据用户问题的具体范围选择相关组件，在 `4.组件详情/<组件名>/` 下按顺序阅读：
   - `README.md` 了解组件职责、输入输出、依赖关系
   - `2.架构.md` 了解内部结构
   - `3.实现步骤.md` 了解实现细节和代码引用

#### 2e. 源码追溯（机制类问题强制执行）

如果是**机制类问题**（第 2a 步分类），**跳过此步就是错误**。无论设计文档看起来多么详细，都必须追溯源码验证。如果是概览类或修改类问题，当设计文档信息不足时才执行此步。

追溯步骤：
   - 根据 `README.md` 中的 **本地仓库路径**（如 `~/sources/{仓库名}/`）浏览项目结构
   - 找到目标操作的 handler 函数（如 operations.ts 中的 handler）
   - 追踪 handler 调用的核心函数/方法，理解实际的数据流和条件分支
   - 如果需要验证 SQL，读取对应的 SQL 查询
   - 结合设计文档和源码给出完整的回答

> 坑：**不要等"感觉设计文档不够充分"再去读源码。** 机制类问题默认就不充分——设计文档的粒度是"功能"，源码的粒度是"if/else 分支"。你读了源码后会惊讶地发现很多实现细节设计文档完全没有覆盖（比如 detail=low 时 SQL 加了一行 `AND chunk_source='compiled_truth'` 这种关键差异）。
>
> 坑：**当答案需要区分"是A、是B、还是两者都有"时，源码是唯一的可靠来源。** 设计文档往往把系统描述为"兼有A和B"，但源码会告诉你 A 走什么路径、B 走什么路径、在什么条件下走哪个。答案的质量差异在此。
>
> 参考示例：`references/mechanism-question-example.md`

### 步骤 3：扩展检索范围（通用知识类问题）
如果以上路径未找到足够信息，进一步查阅：
- `notes/` 下的既有笔记（编程、工具、数据科学等）
- `raw/04-weread/` 中的微信读书划线笔记

### 步骤 4：深度阅读目标文件
选取最相关的页面，使用读取工具获取完整内容。

### 步骤 5：综合与回答
综合信息，回答用户问题。

**双链引用规范**：
- 每当引用某个页面的信息，在文本中使用 `[[页面名称]]` 标注
- 整段引用同一页面：段落首尾各引用一次
- 引用特定原文：使用 Markdown 块引用 `> 引用内容`
- 如果引用的页面 `sources:` 字段指向 `raw/09-archive/`（原始资料已归档），在回答末尾补充说明："原始资料已归档，可查阅 [[摘要-xxx]] 了解详细来源。"

### 步骤 6：高价值内容固化
如果满足以下条件，主动询问用户是否保存为 synthesis：
- 回答超过 2 个段落
- 内容具有分析对比性或总结性

询问话术：
> 这是一个有价值的总结，是否需要我将其保存到 wiki/syntheses/ 目录？

用户同意后，按照 CLAUDE.md 规范创建文件，并在 `wiki/index.md` 的 Syntheses 分类下注册。

用户拒绝后，不创建文件，但**照常执行步骤 7**（记录操作日志），标注 `结果: 即时回答未保存（用户拒绝）`。

### 步骤 7：记录操作日志
无论是否生成 synthesis 页面，查询结束后必须在 `wiki/log.md` 末尾追加：

```markdown
## [YYYY-MM-DD] query | <操作简述>
- **输出**: <引用页面列表或"即时回答未保存">
```

## 强制约束
- **禁止凭记忆回答**：必须先检索知识库
- **禁止过度引用**：同一页面的信息在段落首尾引用一次即可
- **禁止静默回答**：知识库无相关内容时必须声明

## 常见错误（踩过的坑）

### 错误 1：凭训练数据直接回答知识问题
**场景**：用户问"王阳明和张一鸣的哲学有什么共同点"
**错误做法**：直接用自己的知识回答，不搜 wiki
**正确做法**：先搜 wiki/index.md → 发现 [[张一鸣的生活哲学]] 和 [[传习录]] → 基于 wiki 内容回答
**后果**：用户明确不满——"你是直接回答的吗，为什么没搜wiki"
**根因**：触发条件没匹配上，因为用户没说"查一下我的笔记"，只是自然提问
