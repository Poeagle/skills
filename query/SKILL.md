# query 技能

## 核心目标
将用户的提问转化为对本地 Wiki 的深度检索。提取相关页面信息，综合出带有明确引用来源的双链回答。当回答具有高价值时，主动将其固化为知识库的一部分。

## 触发场景
- 用户输入 `/query <问题>`
- 用户用自然语言询问：`"我的笔记里关于 X 是怎么说的"`、`"过去我对 Y 的决策是什么"`、`"查询 Z 相关的知识"`
- 用户提及 wiki、知识库、笔记、记录等关键词

## 降级策略
如果问题属于纯通用知识（如"太阳系有几颗行星"），且 wiki/index.md 中无相关内容：
> 本地知识库中未找到相关内容，以下为通用知识回答：[直接回答]

## 检索与综合流水线

### 步骤 1：查阅全局索引
**永远的第一步**：读取 `wiki/index.md`

在 index.md 中定位与问题相关的：
- Entities（实体）
- Concepts（概念）
- Sources（摘要）
- Syntheses（综合）

### 步骤 2：代码实现类问题的特殊检索路径

如果用户的提问涉及**代码实现、架构设计、组件内部逻辑**（如"某功能是怎么实现的"、"某模块的架构是什么"、"如何修改某处代码"），优先执行以下检索路径：

1. **查找 code-design 文档**：检查 `wiki/code-design/` 下是否有对应仓库的设计文档。例如，关于某个仓库的实现问题，先定位到 `wiki/code-design/{仓库名}/`。
2. **深度阅读组件文档**：
   - 先读 `README.md` 了解整体架构和组件清单
   - 再读相关组件的 `2.架构.md` 了解内部结构
   - 最后读对应组件的 `3.实现步骤.md` 了解实现细节和代码引用
3. **源码追溯**：如果设计文档中的信息不足以回答用户问题，根据 `README.md` 中的 **本地仓库路径**（如 `~/sources/{仓库名}/`）：
   - 直接读取对应源码文件
   - 追溯关键函数的实现
   - 结合设计文档和源码给出完整的回答

### 步骤 3：扩展检索范围（通用知识类问题）
如果以上路径未找到足够信息，进一步查阅：
- `notes/` 下的既有笔记（编程、工具、数据科学等）
- `raw/04-weread/` 中的微信读书划线笔记

### 步骤 4：深度阅读目标文件
选取最相关的页面，使用读取工具获取完整内容。

### 步骤 5：综合与回答
综合信息，回答用户问题。

**双链引用规范**：
- 每当引用某个页面的信息，在文本中使用 `[[页面名称]]` 标注
- 整段引用同一页面：段落首尾各引用一次
- 引用特定原文：使用 Markdown 块引用 `> 引用内容`

### 步骤 6：高价值内容固化
如果满足以下条件，主动询问用户是否保存为 synthesis：
- 回答超过 2 个段落
- 内容具有分析对比性或总结性

询问话术：
> 这是一个有价值的总结，是否需要我将其保存到 wiki/syntheses/ 目录？

用户同意后，按照 CLAUDE.md 规范创建文件，并在 `wiki/index.md` 的 Syntheses 分类下注册。

### 步骤 7：记录操作日志
无论是否生成 synthesis 页面，查询结束后必须在 `wiki/log.md` 末尾追加：

```markdown
## [YYYY-MM-DD] query | <操作简述>
- **输出**: <引用页面列表或"即时回答未保存">
```

---

## 强制约束
- **禁止凭记忆回答**：必须先检索知识库
- **禁止过度引用**：同一页面的信息在段落首尾引用一次即可
- **禁止静默回答**：知识库无相关内容时必须声明
