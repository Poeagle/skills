# ingest 技能

## 核心工作流：Inbox & Archive

你正在维护一个 **LLM Wiki**（Obsidian 知识库）。`raw/` 目录是"待处理收件箱"，`wiki/` 是"编译输出层"。

**目录结构约定：**
- `raw/01-articles/` — 网页剪藏的 Markdown 文章
- `raw/02-papers/` — 论文和 PDF 文献
- `raw/03-transcripts/` — 视频转录文案
- `raw/04-weread/` — 微信读书划线笔记
- `raw/09-archive/` — **已处理文件的归档目录，禁止读取**
- `wiki/sources/` — 资料摘要
- `wiki/entities/` — 实体（人物、公司、工具、产品）
- `wiki/concepts/` — 概念（框架、方法论、理论）
- `raw/04-weread/` 🔒 — 微信读书同步的划线笔记（只读，不在 ingest 中处理，由 query 调用时按需读取）

## 触发逻辑

1. **用户执行 `/ingest`**：扫描 `raw/` 所有子目录（排除 `09-archive/`），找出待处理文件。
2. **用户执行 `/ingest <path>`**：仅处理指定文件。
3. **隐式触发**：用户说"把这个资料摄入知识库"、"导入这篇文章"时，自动执行 ingest。

## 编译流水线

对每个待处理源文件，严格按以下步骤执行：

### 步骤 0：外源采集——先创建原始文件

当源材料来自外部（YouTube链接、网页URL等，而非 raw/ 中已有文件）：

1. **获取内容**：使用 defuddle（网页）或 youtube_transcript_api（YouTube字幕）提取内容
2. **判断分类**：根据内容类型存入对应 raw/ 子目录：
   - YouTube 转录 → `raw/03-transcripts/`
   - 网页文章 → `raw/01-articles/`
   - 论文/PDF → `raw/02-papers/`
3. **保存原始文件**：在对应目录创建 `.md` 文件，加上来源元信息 frontmatter：
   ```markdown
   ---
   source: YouTube
   channel: 频道名
   title: 视频标题
   url: https://...
   retrieved: YYYY-MM-DD
   ---
   ```
   **原始文件命名规则**：文件名使用纯主题描述，**禁止包含创作者名**。创作者信息只存在于 frontmatter 的 `channel:` 字段中。例如：
   - ✅ `美联储深度解析.md`（不是 `小Lin-美联储深度解析.md`）
   - ✅ `唯识宗三部曲.md`（不是 `王利杰-唯识宗三部曲.md`）
4. **确认后继续**：只有原始文件已保存在 `raw/` 中，才进入步骤 1

> **为什么这么做**：wiki 页面的 `sources:` 字段需要指向一个具体的 raw/ 文件路径。没有原始文件，知识溯源链就断了。同时，存档到 09-archive/ 时也需要有文件可移。

### 步骤 1：读取源文件

- **如果是 `.md` 文件**：使用读取工具完整读取内容。
- **如果是 `.pdf` 文件**：使用读取工具尝试提取文本。如果无法提取或内容为空，改为记录文件元信息（文件名、页数）在 sources 页面中。

### 步骤 2：提炼核心并翻译

从源文件中提取：
- **核心主旨**：这段资料讲什么（1-2句话）
- **实体**：人物、公司、工具、产品等具体名词
- **概念**：框架、方法论、理论等抽象名词

如果是非中文内容，则翻译成中文。

### 步骤 3：创建来源摘要

在 `wiki/sources/` 创建 Markdown 文件：

```markdown
---
title: "摘要-文件slug"
type: source
tags: [来源, 原始文件]
sources: [raw/01-articles/xxx.md]
last_updated: YYYY-MM-DD
---

## 核心摘要
[3-5句话的核心总结，提炼源文章最核心的论点，而非罗列所有细节]

## 关键信息
[可选，当源内容较丰富时，使用结构化方式呈现核心内容：
- 使用 ### 子标题划分主题
- 使用表格对比观点/概念
- 使用列表提炼要点
- 使用 > 引用块突出金句]

## 关联连接
- [[EntityName]] — 关联实体
- [[ConceptName]] — 关联概念
```

文件名使用 kebab-case：`摘要-{文件slug}.md`

**Slug 规则**：
- 使用纯主题描述，**禁止包含创作者名**（与原始文件命名规则一致）
- 如果多源同主题有撞名风险，用副标题区分：
  - ✅ `摘要-唯识宗三部曲.md`
  - ✅ `摘要-唯识宗三部曲-认知科学视角.md`
  - ❌ `摘要-王利杰-唯识宗三部曲.md`

### 步骤 4：知识网络化（实体/概念页面）

对于步骤 2 提取的每个实体和概念：

**目标目录：**
- 实体 → `wiki/entities/`
- 概念 → `wiki/concepts/`

**处理逻辑：**
1. 页面不存在 → 按照 CLAUDE.md 的 Frontmatter 规范创建新页面
2. 页面已存在 → 读取现有内容，**增量合并**新信息
3. **发现冲突** → **立即暂停**，向用户报告冲突内容，询问处理方式后再继续

**实体判定原则**——什么值得建页面：
- ✅ **值得创建**：该人物/机构本身就是知识体系的一部分。用两个问题验证：
  1. 这个人/机构**本身**是知识体系的分析对象吗？（如 [[美联储]]、[[字节跳动]]）
  2. 这个人的思想/作品**独立于其讲解行为**具有知识价值吗？（如 [[AndrejKarpathy]] —— 其论文和项目是知识来源）
  两个满足其一即可。
- ❌ **不值得创建**：内容创作者、博主、UP主等"知识传递者"——他们是管道而非知识本身。即便内容中包含其原创性洞见（如用 AI 架构验证唯识论），只要其核心身份是"讲解者"而非"被分析对象"或"独立知识生产者"，就不建实体。来源摘要的 `sources:` 字段记录足矣。
- ⚠️ 注意区分："原创洞见的提出者" ≠ "值得建实体"。如果该人物的影响力完全绑定在某一套讲解体系中（如频道主讲人），仍然属于知识管道类别。

**页面模板：**

```markdown
---
title: "页面名称"
type: entity | concept
tags: [标签]
sources: [关联的源文件]
last_updated: YYYY-MM-DD
---

## 定义
[对该实体/概念的定义，1-3句话说清核心内涵]

## 关键信息
[从源文件中提取的详细信息，必须结构化呈现：
- 使用 ### 子标题划分不同维度（如类型对比、核心机制、应用场景）
- 使用表格（|  |  |）做对比，适用于：概念对比、阶段划分、维度分析
- 使用有序列表描述流程/步骤
- 使用无序列表提炼要点
- 使用 > 引用块突出核心金句或洞见
- 避免连续超过 5 行的纯段落文字
- 宁可分段、分条，也不要一大块文字]

## 关联连接
- [[摘要-source-slug]] — 来源
- [[RelatedEntity]] — 相关实体
```

### 页面知识洁癖——源头清理规则

在步骤 3 和步骤 4 创建/更新页面时，必须执行以下清理：

**规则 A：剥离创作者署名**
来源摘要正文和概念页中，不得出现创作者（博主/UP主/主讲人）的姓名、别称、背景介绍或风格评价。原始出处通过 frontmatter 的 `sources:` 字段追溯即可。

**规则 B：引用块去归属**
当从引用块中移除创作者署名（如 `——王利杰`）时：
- 若引文表达的是**通用洞见**（不依赖说话者身份），保留引文，仅去除 `——名字`
- 若引文是**个人经历叙述**（如"我修行五年..."），改为间接转述或删除
- 若去除署名后引文出现无所指代的代词（他/她/他们/其），改写为明确指代或转述

**规则 C：代词残留检查**
从页面中移除创作者信息后，扫描全文中的第三人称代词（他、她、他们、其、自己），确认每个代词都有明确的指代对象。若指代悬浮，替换为所指代的具体概念名或实体名。

**规则 D：禁止元评价**
来源摘要和概念页中不应出现对创作者个人的描述性评价，包括但不限于：
- 风格评价（"招牌式的幽默风格"、"以...著称"）
- 传播力评价（"深入浅出"、"妈妈也能听懂"）
- 个人背景（"前华尔街从业者"、"多年修行经验"）
聚焦内容本身，不评价格式和渠道。

### 步骤 5：更新全局注册表

**更新 `wiki/index.md`：**
按照 CLAUDE.md 规定的格式，将新增页面添加到对应分类下。

**更新 `wiki/log.md`：**
追加操作日志（Append-only）：
```markdown
## [YYYY-MM-DD] ingest | 操作简述
- **变更**: 新增 [[PageName]]; 更新 [[index.md]]
- **冲突**: 无（或: 冲突 [[ConflictingPage]], 已暂停等待决策）
```

### 步骤 5.5：重命名检修（仅在重命名 source 文件时执行）

如果本次操作涉及 source 文件重命名（例如修正slug），在更新 index.md 之后、归档之前，必须执行：

1. **扫描旧名引用**：用 grep 搜索 `wiki/` 下所有文件中是否仍有 `[[旧文件名]]`（注意不含路径和后缀的 wikilink 格式）
2. **逐一修正**：对每个残留引用，更新为 `[[新文件名]]`
3. **再次确认**：二次 grep 确认零残留
4. **更新主目录**：确认 index.md 中的引用已同步

> 为什么需要这一步：Obsidian wikilink 用的是文件名（不含 `.md`），source 文件改名后所有概念/实体页面的 `关联连接` 中的引用都会失效，变成死链。

### 步骤 6：归档源文件

在确认以下全部完成后，将源文件移动到 `raw/09-archive/` 目录：
- sources 页面已创建
- 实体/概念页面已创建或更新
- index.md 已更新
- log.md 已更新

**绝对禁止修改源文件内部的文字。**

## 冲突处理流程

当发现新旧知识冲突时：

1. **暂停**：停止当前 ingest 流程
2. **报告**：向用户说明冲突内容（哪个页面、冲突点是什么）
3. **询问**：请用户选择处理方式：
   - A) 保留新旧两者，标注为"知识冲突"
   - B) 用新知识覆盖旧知识
   - C) 放弃本次 ingest
4. **继续**：根据用户选择继续或终止

## 注意事项

- 绝对不读取 `raw/09-archive/` 下的任何文件
- 所有 wiki 页面必须包含 `## 关联连接` 区域，不能产生孤岛页面
- 使用简体中文编写所有内容
- 实体命名使用 TitleCase，来源/综合使用 kebab-case
