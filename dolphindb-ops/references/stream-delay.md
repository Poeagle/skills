# 流计算延迟/堆积排查

> 触发: 流引擎队列深度持续增长、数据到达延迟超 SLA、消费端落后

## 规则

### 计算瓶颈
条件: 引擎 queueDepth 持续增长 + CPU 高
处置: 取消抢占 CPU 的大查询
根因: 简化聚合函数；减少 keyColumn 基数；考虑预聚合

### IO 瓶颈
条件: 引擎 queueDepth 持续增长 + IO 高
处置: 清理数据库缓存释放资源给流引擎
根因: 输出表改用异步持久化；减少持久化频率

### 窗口/分组过大
条件: 某引擎 queueDepth 显著高于其他引擎
处置: 缩小 timeSeriesEngine 窗口到实际需要的最小值；按 hash 分流到多个引擎实例

### 消费回调慢
条件: 流订阅状态中 subWorkers lastMsgId 停滞
处置: 增加流消费线程数（通过配置或运行时调整）
根因: 优化下游写入/callback 函数

### 上游数据源问题
条件: 发布端 lastMsgId 停滞
处置: 检查数据源连接、网络延迟

### 跨节点网络延迟
条件: 发布端和订阅端不在同一节点，且订阅端 lastMsgId 持续落后发布端
处置: 检查两节点间网络延迟（`ping`）和带宽利用率；若延迟高（>5ms），考虑将订阅端迁移到同机房/同机架；若带宽瓶颈，检查是否有其他大流量任务挤占网络

### GC 停顿
条件: 引擎 queueDepth 持续增长 + CPU/IO 正常
处置: 检查 DDB 运行日志中 GC 相关记录（搜索 "GC" 关键字）

### 输出表无限增长
条件: 流引擎内存占用持续增长
处置: 对输出表设置数据保留策略（按天数保留）

## 验证
- 各引擎 queueDepth 趋势是否收敛
- subWorkers lastMsgId 是否追上
- 消费节点 CPU 利用率正常
- 引擎延迟回到 SLA 内
