## 产品与架构概览

DolphinDB 是一款高性能分布式时序数据库，采用 shared-nothing 分布式架构，支持多模态存储、ACID 事务，并通过 MVCC 实现多版本并发控制。系统在物理部署之上构建逻辑文件系统，由 controller 管理元数据，分区数据按逻辑划分存储在不同 data node 上，屏蔽底层物理路径细节。其分布式能力覆盖高可用、在线恢复、分布式事务和异步复制等场景。

## 运行模式

- 单实例模式
- 集群模式

常见集群部署场景：

- 伪分布式集群
- 伪高可用集群
- 高可用集群

## 节点角色

### controller

- 收集 agent、data node、compute node 的心跳
- 监控节点在线状态
- 管理分布式文件系统元数据和事务
- 是集群核心管理节点

### agent

- 执行 controller 下发的启动和关闭 data node、compute node 的命令
- 每台物理服务器有且仅有一个 agent

### data node

- 负责数据存储、查询以及部分计算任务
- 每台物理服务器可以配置多个 data node

### compute node

- 只负责计算，不存储数据
- 常用于流计算、分布式关联、机器学习等场景

## 节点通信与心跳

- 集群模式下，agent、data node、compute node 默认通过 UDP 广播通信
- 也可以通过设置 `lanCluster=0` 启用 TCP 广播
- 这些节点每秒向 controller 发送一次心跳
- 如果 controller 连续 3 秒未收到某节点心跳，则认为该节点已掉线

## 高可用机制

### 数据高可用

- 通过多副本机制在不同服务器上保存副本
- 通过两阶段提交保证副本之间以及数据和元数据之间的强一致性

### 元数据高可用

- 多个 controller 组成 Raft 组
- 通过 Raft 实现 controller 高可用

### 客户端高可用

- 客户端可配置多个 data node site，支持断连切换
- 流表读写场景可自动感知 leader 变化并切换连接

## 分布式事务

- 对分布式表的写入、更新、删除都可视为分布式事务
- 系统通过 2PC、Redo Log、Cache Engine 和 MVCC 实现事务管理
- 多副本场景下，controller 维护副本元数据和版本信息
- 节点恢复后可通过在线恢复机制修复落后副本
- controller 故障时可依靠 Raft 保证事务状态同步
- data node 故障恢复后可请求事务决议并进行副本恢复

## 运维分析视角

从运维角度分析 DolphinDB 问题时，应优先明确：

- 问题涉及哪些节点角色
- 涉及单实例还是集群模式
- 节点当前是否在线、心跳是否正常
- 是否存在配置异常、通信异常或资源瓶颈
- 是否影响数据分区、副本、元数据、事务或恢复流程

常见根因可归纳为：

- 节点状态问题
- 资源问题
- 配置问题
- 通信问题
- 数据问题
- 副本问题
- 元数据问题
- 事务问题
