# 作业管理问题诊断

> 触发: 批处理作业长时间排队、作业执行失败、作业无法取消、作业队列堵塞

## 规则

### 批处理作业长时间排队
条件: 查看最近批处理作业显示大量作业 startTime 为空（还在排队）
处置:
1. 查看作业队列深度统计
2. 检查 maxBatchJobWorker 配置是否过小
3. 查看是否有长时间运行的作业阻塞队列
4. 考虑取消低优先级或异常作业释放资源
根因: 批处理作业工作线程数不足；存在长时间运行的作业占用资源；作业提交速度超过处理速度

### 批处理作业执行失败
条件: 查看最近批处理作业显示作业 errorMsg 非空
处置:
1. 查看具体作业状态
2. 查看作业中间消息
3. 分析 errorMsg 确定失败原因（权限、资源、语法等）
4. 根据错误类型参考对应 category（如 oom、security-guide 等）
根因: 权限不足、内存溢出、磁盘空间不足、语法错误、数据异常

### 批处理作业无返回结果
条件: 作业已完成但无法获取返回结果
处置:
1. 尝试获取作业返回对象
2. 检查 batchJobDir 目录下的 `<jobId>.obj` 文件是否存在
3. 检查磁盘空间是否充足
根因: 返回对象过大导致序列化失败；磁盘空间不足；batchJobDir 配置错误

### 同步作业无法取消
条件: 取消同步作业失败或作业仍在运行
处置:
1. 检查节点连接数是否已满（maxConnections）
2. 尝试从其他节点取消作业
3. 检查作业队列是否堵塞导致取消命令无法执行
4. 使用 GUI 的取消按钮（以 urgent 优先级发送）
根因: 连接数用完；作业队列堵塞；作业处于不可中断状态

### 作业队列堵塞
条件: 作业队列深度持续增长，或同步作业队列显示大量作业排队
处置:
1. 查看排队的同步作业
2. 识别并取消异常的大查询或长时间运行的作业
3. 检查是否有死锁或资源耗尽（参考 system-hang）
4. 检查 worker 线程数配置（workerNum、webWorkerNum）
根因: worker 线程数不足；存在死锁；大量并发作业提交；单个作业耗时过长

### 批处理作业工作线程不足
条件: 批处理作业队列深度高，但系统资源（CPU/内存）未打满
处置:
1. 检查 maxBatchJobWorker 配置值
2. 查看 BatchJobWorker 实际数量和状态
3. 调大 maxBatchJobWorker（建议设为 CPU 核数的 1-2 倍）
4. 重启节点使配置生效
根因: maxBatchJobWorker 配置过小，无法充分利用系统资源

### Worker 线程被批处理作业占满
条件: 同步作业响应慢，线程诊断显示大量 BatchJobWorker 在运行
处置:
1. 查看 Worker 和 BatchJobWorker 数量及状态
2. 减少批处理作业的并发数量
3. 或者增加 workerNum 配置，为同步作业预留更多线程
4. 考虑使用计算节点隔离批处理作业
根因: 批处理作业过多，占用了同步作业的 Worker 线程

### 作业优先级配置不当
条件: 高优先级作业仍然等待很久，或低优先级作业长时间得不到执行
处置:
1. 查看最近作业和同步作业的优先级分布
2. 检查用户的最高优先级设置
3. 提交批处理作业时合理设置优先级（0-9）
4. 避免所有作业都设置高优先级导致优先级失效
根因: 优先级设置不合理；所有作业优先级相同导致按提交顺序执行

### 作业并行度配置不当
条件: 作业有大量子任务但执行很慢，或单个作业占用过多资源
处置:
1. 查看作业的并行度设置
2. 对于子任务多的作业，适当提高并行度
3. 对于资源密集型作业，降低并行度避免资源争抢
4. 注意并行度只是时间片单位，实际线程数由系统动态分配
根因: 并行度设置过低导致资源利用不足；并行度设置过高导致资源争抢

### 作业子任务分布不均衡
条件: 某些节点负载很高，其他节点空闲
处置:
1. 查看各节点的同步作业和批处理作业分布
2. 检查数据分区在各节点的分布是否均衡
3. 检查各节点的计算资源配置是否一致
根因: 数据分区分布不均；节点配置不一致；作业提交方式导致负载倾斜

### 定时作业执行异常
条件: 定时作业状态异常或长时间未执行
处置:
1. 查看定时作业列表和下次执行时间
2. 检查定时作业的调度频率是否合理
3. 检查是否有定时作业执行时间过长导致下次调度延迟
4. 查看日志中定时作业的错误信息
根因: 定时作业执行时间超过调度间隔；定时作业代码错误；系统资源不足

### 作业结果文件丢失
条件: 批处理作业完成但无法找到结果文件
处置:
1. 检查 batchJobDir 配置的目录路径
2. 查看 `<batchJobDir>/batchJob.log` 日志文件
3. 检查 `<batchJobDir>/<jobId>.msg` 和 `<jobId>.obj` 文件
4. 检查磁盘空间和文件权限
根因: batchJobDir 配置错误；磁盘空间不足；文件权限问题；节点重启导致内存中的作业信息丢失

### 节点重启后作业信息丢失
条件: 节点重启后无法查看重启前的作业
处置:
1. 查看 `<batchJobDir>/batchJob.log` 获取历史作业信息
2. 检查 `<batchJobDir>` 下的作业结果文件
3. 注意查看最近作业只返回当前启动后的作业信息
根因: 最近作业列表不保存重启前的作业信息，需要从磁盘文件查看

## 验证
- 作业队列深度恢复正常
- 作业执行成功率提升
- 作业等待时间缩短
- 各节点负载均衡
- 系统资源利用率合理

## 作业类型说明

### 同步作业（交互式作业）
- **提交方式**: GUI、Console、Terminal、VSCode 插件、API、Web notebook
- **特点**: 阻塞客户端连接，返回前无法发送新作业
- **并发限制**: 取决于 maxConnections（连接数）和 workerNum/webWorkerNum（工作线程数）
- **优先级**: 默认为 min(4, 用户最高优先级)
- **查询方式**: 查看当前同步作业列表
- **取消方式**: 取消指定同步作业

### 批处理作业（异步作业）
- **提交方式**: 异步提交接口（支持指定优先级和并行度）
- **特点**: 后台执行，不阻塞客户端
- **并发限制**: 取决于 maxBatchJobWorker
- **优先级**: 默认为 4，可在提交时指定（0-9）
- **并行度**: 默认为 2，可在提交时指定
- **查询方式**: 查看最近批处理作业列表、查看单个作业状态、获取作业消息
- **取消方式**: 取消指定批处理作业
- **结果获取**: 获取作业返回对象

### 定时作业
- **提交方式**: 定时调度接口
- **特点**: 按计划周期性执行
- **优先级**: 固定为 4，无法修改
- **查询方式**: 查看定时作业列表
- **取消方式**: 删除定时作业

### 流数据作业
- **提交方式**: 订阅流表
- **特点**: 持续运行，处理实时数据
- **相关 category**: `stream-delay`

## 作业调度机制

### 优先级调度
- **优先级范围**: 0-9，数值越高优先级越高
- **调度策略**: 多级反馈队列，高优先级作业优先获得资源
- **动态降级**: 作业执行完一个时间片后，如果有低优先级作业等待，则降低一级优先级
- **防饥饿**: 低优先级作业通过动态降级机制最终会被调度

### 并行度
- **含义**: 单个作业最多同时使用的线程数（时间片单位）
- **默认值**: 2
- **实际线程数**: 系统根据资源情况动态分配，可能超过并行度设置
- **示例**: 并行度为 2 的作业有 100 个子任务，需要 50 轮调度完成

### 子任务分解
- **SQL 查询**: 按数据库分区分解，每个分区一个子任务
- **Map-Reduce**: 按数据源分解
- **计算下推**: SQL 查询的子任务在数据节点本地执行（计算存储耦合）

## 性能优化建议

### 配置优化
1. **worker 线程数**: `workerNum` 建议设为 CPU 核数，`webWorkerNum` 根据 Web 访问量设置
2. **批处理线程数**: `maxBatchJobWorker` 建议设为 CPU 核数的 1-2 倍
3. **连接数**: `maxConnections` 根据并发用户数设置，建议 100-1000
4. **作业目录**: `batchJobDir` 建议设置在高速磁盘上

### 作业提交优化
1. **合理设置优先级**: 重要作业设高优先级，避免所有作业都设高优先级
2. **合理设置并行度**: 子任务多的作业提高并行度，资源密集型作业降低并行度
3. **避免大量并发**: 控制同时提交的作业数量，避免队列堵塞
4. **使用批处理**: 对于不需要实时反馈的任务，使用异步提交而非同步执行

### 数据分区优化
1. **分区均衡**: 确保各分区数据量均衡，避免数据倾斜
2. **分区分布**: 确保分区在各节点均匀分布
3. **分区粒度**: 分区数量建议为节点数 × CPU 核数的 10-100 倍

### 计算节点使用
- **适用场景**: 计算密集型操作（流计算、分布式关联、机器学习、数据分析）
- **优势**: 隔离计算和存储资源，避免相互影响
- **配置**: 从 1.30.14/2.00.1 版本开始支持

## 注意事项

1. **作业取消延迟**: 取消作业通过设置 Cancel 标志实现，不是立即生效，可能有延迟
2. **连接数限制**: 同步作业数量受 maxConnections 限制，连接用完无法提交新作业
3. **重启丢失**: 查看最近作业只返回当前启动后的作业，重启前的作业需从磁盘文件查看
4. **结果文件**: 批处理作业结果保存在 batchJobDir，定期清理避免占用过多磁盘空间
5. **跨节点操作**: 可在其他节点执行作业管理操作
6. **紧急取消**: GUI 取消按钮以 urgent 优先级发送，可在队列堵塞时使用
