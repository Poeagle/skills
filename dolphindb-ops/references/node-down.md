# 节点宕机排查（分类分流）

> 触发: 节点不可达、心跳超时、集群中节点状态异常

## 排查顺序

1. 先查看进程状态、端口监听状态，判断进程是否存在及是否正常监听。
2. 再结合内核日志、运行日志和错误日志判断是 OOM、崩溃、配置错误还是系统阻塞。
3. 只有在进程状态正常、端口监听正常时，才把 `heartbeat`、`node offline`、`connection refused` 这类通用症状归入心跳或网络分支。

## 规则

### 进程暂停 (T / t)
条件: 进程状态显示 `State: T` 或 `State: t`
处置: 只要当前进程状态是 T/t，就优先按"暂停态"处理；即使内核日志中存在历史 segfault，也不要先归类为崩溃或直接建议重启。优先恢复暂停态进程，再复查进程状态、端口监听和节点在线状态。只有在恢复动作本身成功执行后仍失败、进程仍保持 T/t、或恢复后立即再次进入停止态时，才继续升级到人工排查或节点重启。

### 进程阻塞 (D)
条件: 进程状态显示 `State: D`
处置: 视为不可中断阻塞。优先排查磁盘、IO、NFS、内核阻塞或设备异常，结合磁盘状态、IO 状态、内核日志判断根因。不要先建议 kill 或重启。

### 僵尸或已退出 (Z)
条件: 进程状态显示 `State: Z`
处置: 视为进程已不可用，排查父进程回收和节点拉起链路。若需要恢复，通过平台 API 启动节点；不要把 Z 态当作可通过 SIGCONT 恢复的暂停态。

### 端口未监听但进程存在
条件: 能定位到 DDB 进程，但端口无监听结果
处置: 优先判断为半启动、假存活或启动后挂起，不要直接归类为心跳超时。检查运行日志、错误日志和配置文件，必要时参考 `startup-failure`。

### OOM Kill
条件: 内核日志出现 oom_kill_process / killed process / Out of memory
处置: 检查内存配置，调整后重启节点；参考 `oom` 深入排查

### 进程崩溃 (segfault / core dump)
条件: 内核日志出现 segfault / core dump；且 DDB 进程不存在或已经退出
处置: 通过平台 API 启动节点；参考 `crash` 深入分析崩溃原因

### 磁盘满
条件: 磁盘空间使用率 >95%
处置: 清理空间后重启节点
根因: 排查空间大户（查看节点磁盘使用情况和数据目录占用）

### 心跳超时
条件: 集群性能指标中节点 isRunning=false，但 SSH 可达、进程状态为 R/S/I 等可运行状态，且端口监听正常
处置: 检查网络延迟、NTP 同步状态，无问题则重启节点
根因: 调整心跳超时参数；排查网络问题

### 配置错误
条件: DDB 错误日志中有 startup 相关 error/fatal，或进程存在但端口未监听且日志显示启动失败
处置: 修正配置后重启节点；参考 `startup-failure`
根因: 检查 dolphindb.cfg 中的启动参数

### License 过期
条件: DDB错误日志出现 `invalid license`
处置: 更新 license 文件；参考 `license-update`

### 机器级故障
条件: SSH 不可达 / 所有检查超时
处置: 联系基础设施团队检查物理机（硬件/电源/内核 panic）

## 验证
- 若之前是 T/t 暂停态，确认进程状态已恢复为可运行态，且最近未再次进入暂停态
- 若之前是"进程存在但端口未监听"，确认端口已恢复监听且节点重新上线
- 确认节点状态已恢复为 running
- 检查副本恢复进度是否正常
- 重启后日志是否有新的异常
- 系统内存和磁盘空间是否稳定在安全水位
